# 实践提交清单

这篇文档用于训练或比赛最后检查。目标是确保不只是“机器人跑到了”，还留下了能证明你完成任务的截图和参数记录。

## 1. 启动前检查

确认工作空间：

```bash
cd ~/ros2_ws
source install/setup.bash
```

确认 ROS 版本：

```bash
ros2 --version
```

确认仿真时间：

```bash
ros2 param get /rviz2 use_sim_time
```

如果是 Gazebo 仿真，Nav2、SLAM、RViz 都应使用 `use_sim_time:=True`。

## 2. 必须成功的运行现象

Gazebo：

- 机器人实体出现。
- 场景/障碍物出现。
- 机器人能响应速度或导航指令移动。

RViz：

- 能看到地图。
- 能看到机器人模型。
- 能看到雷达扫描。
- 能看到全局代价地图。
- 能看到局部代价地图。
- 下发目标后出现全局路径。
- 机器人最终到达目标附近。

## 3. 必须保留的截图

### 3.1 启动成功截图

截图内容：

- Gazebo 窗口中机器人和预设场景。
- RViz 中地图、机器人、雷达。

用途：

- 证明“加载预设导航场景”完成。

### 3.2 导航运行截图

截图内容：

- RViz 中 `Nav2 Goal` 后生成的路径。
- 机器人正在沿路径运动。
- global/local costmap 可见。

用途：

- 证明“启动导航功能”完成。

### 3.3 到达终点截图

截图内容：

- Gazebo 或 RViz 中机器人到达目标位置。
- 如果 RViz 有状态反馈，尽量截到导航成功状态。

用途：

- 证明“机器人按要求抵达终点”完成。

### 3.4 参数修改截图

截图内容：

- `nav2_params.yaml` 中你改过的参数。
- 推荐截这些区域：
  - `global_costmap`
  - `local_costmap`
  - `inflation_layer`
  - `robot_radius` 或 `footprint`
  - `planner_server`
  - `controller_server`

用途：

- 证明“调整导航参数”完成。

## 4. 推荐记录的命令

启动 Gazebo：

```bash
ros2 launch <package_name> <gazebo_launch_file>.launch.py
```

启动 SLAM：

```bash
ros2 launch slam_toolbox online_async_launch.py use_sim_time:=True
```

启动 Nav2：

```bash
ros2 launch nav2_bringup navigation_launch.py use_sim_time:=True params_file:=<nav2_params.yaml>
```

检查 TF：

```bash
ros2 run tf2_ros tf2_echo map odom
ros2 run tf2_ros tf2_echo odom base_link
```

检查传感器：

```bash
ros2 topic echo /scan --once
ros2 topic echo /odom --once
```

## 5. 答题报告建议写法

可以按这个结构写：

1. **场景加载**：说明启动了哪个 Gazebo/RViz/launch 文件。
2. **导航启动**：说明使用了 Nav2、SLAM 或 AMCL。
3. **参数调整**：列出修改了哪些参数，以及为什么改。
4. **导航结果**：说明机器人从起点到终点，是否避障成功。
5. **截图说明**：每张图对应一个任务要求。

参数说明示例：

```text
为避免机器人贴墙行驶，调整 global/local costmap 的 inflation_radius；
为使碰撞检测更接近机器人真实外形，配置 local_costmap 的 footprint；
为提升局部避障稳定性，调整 controller_server 中的速度上限。
```

## 6. 最后复核

提交前确认：

- 已截图参数文件。
- 已截图 RViz/Gazebo 成功运行。
- 目标点确实到达，不只是路径生成。
- 截图能看清关键参数名和值。
- 答题报告中每张图有对应说明。

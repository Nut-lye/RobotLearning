# Nav2 通信与生命周期

> 阅读位置：Nav2 理论第一层。
>
> 前置建议：先跑一遍 `实践操作/Turtlebot3仿真导航快速跑通.md`，知道导航任务会持续一段时间。
>
> 本章目标：理解 Nav2 为什么用 Action 表达导航任务，以及为什么核心节点要通过生命周期统一管理。

## 1. 为什么导航不能只用 Service

普通 ROS 2 Service 适合短任务：

```text
发送请求 -> 立即得到响应
```

例如：

- 查询一个参数。
- 请求保存地图。
- 调用一个瞬间完成的计算。

但导航不是瞬间完成的。机器人从当前位置走到目标点，可能需要几十秒甚至几分钟，中途还可能：

- 遇到障碍物。
- 重新规划路径。
- 被取消任务。
- 失败后进入恢复动作。

所以 Nav2 使用 Action 表达这种长时间运行任务。

## 2. Action 的三个核心部分

Action 通信包含：

1. **Goal**：客户端发送目标。
2. **Feedback**：服务端执行过程中持续反馈进度。
3. **Result**：任务结束后返回结果。

以“去厨房”为例：

- Goal：发送厨房的地图坐标。
- Feedback：持续返回当前距离目标还有多远、是否仍在执行。
- Result：到达、失败、取消或中止。

如果没有 Action，机器人在移动过程中就很难被监控、取消或恢复。

## 3. Nav2 中常见 Action

Nav2 中的行为树会调用多个 Action Server：

- `NavigateToPose`：导航到单个目标点。
- `ComputePathToPose`：规划从当前位置到目标点的路径。
- `FollowPath`：沿着路径运动。
- `Spin`：原地旋转恢复。
- `BackUp`：后退恢复。
- `Wait`：等待。

这些 Action 通常不是由用户直接一个个调用，而是由行为树统一调度。

## 4. 为什么需要生命周期节点

普通节点一启动就可能开始发布、订阅和执行逻辑。但导航系统很复杂，如果各节点无序启动，可能出现：

- 控制器已经输出速度，但雷达还没准备好。
- costmap 还没拿到 TF，就开始计算。
- planner 还没加载参数，就收到规划请求。
- 某个关键节点失败，但其他节点仍继续运行。

生命周期节点用状态机解决这个问题。

## 5. 生命周期状态

常见状态：

```text
Unconfigured -> Inactive -> Active -> Finalized
```

含义：

- `Unconfigured`：节点刚创建，还没有加载完整参数和资源。
- `Inactive`：节点已经配置好，但还没有真正开始工作。
- `Active`：节点正式运行，发布数据、接收任务。
- `Finalized`：节点结束。

典型流程：

1. 配置阶段：加载参数、申请资源、初始化插件。
2. 激活阶段：开始发布 costmap、接收 Action、输出控制。
3. 降级阶段：发生异常时停止执行，避免机器人继续运动。

## 6. 生命周期管理器的作用

Nav2 中的 `lifecycle_manager` 负责统一管理多个生命周期节点。

它的作用可以理解为“总电源开关 + 安全检查员”：

1. 按顺序配置节点。
2. 确认节点都准备好。
3. 统一激活。
4. 节点出问题时进行降级或关闭。

所以，当机器人“不动”时，除了看 `/cmd_vel` 和 TF，也要看生命周期状态。

检查命令：

```bash
ros2 lifecycle nodes
```

如果核心节点没有进入 active，导航不会正常执行。

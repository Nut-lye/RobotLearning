## 本章在导航系统中的位置

行为树中的节点彼此解耦，不能像普通函数一样直接互相传参。黑板就是它们共享数据的地方。

本章对应链路：

```text
GoalUpdater / ComputePathToPose / FollowPath 等 BT 节点
  -> 黑板 key-value 数据
  -> goal、path、updated_goal 等变量流转
```

如果只是普通到点导航，知道黑板概念即可；如果要改动态追踪或写行为树插件，就需要认真理解它。

### 黑板 (Blackboard) 

#### 1. 概念定义

在 ROS 2 Nav2 的行为树（Behavior Tree）架构中，“黑板”（Blackboard）是一个依附于行为树实例的局部键值对（Key-Value）数据库。它专门用于处理各个独立节点之间的数据交换与状态共享。

黑板本质上是一个**全局的键值对数据库（字典）**，所有节点都有权限在这块黑板上“写字”和“擦字”

#### 2. 架构意义与解决的痛点

行为树框架的核心设计理念是“节点的高度解耦”。这意味着：

- 每一个节点（如规划节点 `ComputePathToPose` 或控制节点 `FollowPath`）在 C++ 底层都是完全独立的类。
- 节点之间互不可见，无法通过传统的函数传参（Arguments/Parameters）方式直接递交数据。

为了在互不依赖的节点之间建立数据流，系统引入了黑板机制。黑板充当了共享内存中枢，上游节点将运算结果写入黑板，下游节点从黑板读取所需数据，从而实现了系统的高内聚与低耦合。

#### 3. 数据流转原理（基于 XML 语法）

在行为树的 XML 配置文件中，节点的数据接口被称为**端口（Ports）**。带有大括号 `{}` 的字符串代表该端口与黑板上特定键名（Key）的映射关联。

结合前文的“动态追踪”配置，黑板的数据流转过程如下：

1. **数据写入（Output Port）：** `GoalUpdater` 节点作为上游数据源，持续接收 ROS 话题发来的新坐标。它通过输出端口 `output_goal`，将最新坐标写入黑板上的键 **`{updated_goal}`**。
2. **数据读取与加工（Input/Output Port）：** `ComputePathToPose`（全局规划器）通过输入端口 `input_goal`，从黑板上提取 **`{updated_goal}`** 对应的值进行路径计算。计算完成后，它通过输出端口将生成的长路径写入黑板的 **`{path}`** 键。
3. **数据重构：** `TruncatePath`（路径截断器）从黑板读取 **`{path}`**，进行裁剪算法处理后，将缩短后的新路径写入黑板的 **`{truncated_path}`** 键。
4. **数据消费（Input Port）：** `FollowPath`（局部控制器）作为末端消费者，它的唯一依赖是从黑板上读取 **`{truncated_path}`**。一旦读取成功，它便根据该轨迹输出底盘电机的物理控制指令。

#### 4. 黑板机制的工程优势

- **模块化即插即用：** 只要保证黑板上的键名（Key）和数据类型一致，开发者可以随意增删或重排 XML 中的节点，而不需要修改任何 C++ 源码。例如，若移除 `TruncatePath` 节点，只需让 `FollowPath` 直接读取 `{path}`，系统依然能够正常运作。
- **全局状态共享：** 黑板不仅用于传递路径和坐标，还可以存储系统的全局状态（如机器人电量、当前运动速度、恢复操作重试次数等），供整棵行为树中的任意分支节点进行条件判断（Condition Check）。
